# Мобильный контроллер игрока

Этот документ описывает не web-версию игры, а телефонный интерфейс игрока. Unity остаётся главным приложением и авторитетным сервером, а телефон только показывает состояние своего персонажа и отправляет намерения: двигаться, атаковать, сменить оружие, подтвердить переход и так далее.

Цель: сделать полноценный локальный сайт, который открывается по QR-коду и работает как постоянный клиент игрока. Это не должен быть один большой `index.html`; интерфейс нужно разделить на нормальные страницы/экраны, стили, скрипты и API-клиент.

## Целевой пользовательский сценарий

1. GM запускает локальный сервер.
2. На экране игроков показывается QR-код.
3. Игрок открывает сайт с телефона.
4. Сайт показывает регистрацию: имя и фото.
5. Игрок отправляет регистрацию.
6. На player view появляется подключившийся игрок.
7. GM входит в выбор кампании.
8. Игроки видят лобби и выбранную/доступную кампанию вместе с GM.
9. В лобби игроки занимают места сверху, но сначала они ещё не готовы: у них есть только имя и фото, но нет персонажа.
10. Портреты таких игроков в лобби отображаются серыми.
11. На телефоне игрок получает выбор: создать персонажа или импортировать существующего.
12. При импорте существующего персонажа он привязывается к игроку. Аватар персонажа заменяется на фото игрока или игроку даётся выбор, оставить старый портрет или использовать новое фото.
13. При создании персонажа открывается полноценный редактор персонажа на телефоне.
14. Когда игрок выбрал или создал персонажа, он становится готов.
15. Когда все игроки готовы, GM запускает игру/бой.

## Состояния телефона

- `register` - первичная регистрация: имя и фото.
- `waiting_for_gm` - игрок подключён и ждёт действий GM.
- `campaign_lobby` - выбор/ожидание кампании вместе с GM.
- `character_choice` - создать персонажа или импортировать существующего.
- `character_editor` - полноценный редактор персонажа.
- `character_import` - выбор существующего персонажа.
- `ready_lobby` - персонаж выбран, игрок готов.
- `exploration` - обычная кампания вне боя.
- `combat` - бой.
- `inventory` - инвентарь.
- `character_settings` - настройка персонажа, оружия, способностей, прокачки.

При перезагрузке страницы телефон не должен возвращаться на регистрацию, если игрок уже подключён. Клиент должен восстановить сессию по локально сохранённому `playerId`/токену и запросить актуальное состояние у Unity.

## Что уже реализовано

- `WebServerManager` автоматически создаётся до загрузки сцены и поднимает локальный TCP/HTTP сервер на порту `8080`.
- Web-клиент лежит в `Assets/StreamingAssets/WebClient`.
- Главный экран для игроков показывает QR-код с адресом `http://<local-ip>:8080`.
- Телефон может открыть страницу, выбрать фишку, ввести имя, описание, загрузить фото и подключиться к лобби.
- При подключении создаётся персонаж через `CampaignGameSession.AddCharacterPlayer`.
- Экран ожидания телефона опрашивает `/api/lobby/status`.
- После старта кампании телефон переходит на игровой экран.
- Уже есть D-pad для движения фишки.
- Уже есть подтверждение перехода между картами.
- Уже есть список ближайших врагов и простая кнопка атаки.

## Текущие API

- `GET /api/lobby/tokens` - список доступных фишек для выбора в лобби.
- `GET /api/image?path=...` - отдача изображения по закодированному пути.
- `POST /api/lobby/join` - подключение игрока, сохранение фото и создание персонажа.
- `GET /api/lobby/status?playerId=...` - ожидание старта игры.
- `GET /api/game/state?playerId=...` - состояние игрока, переходы и ближайшие враги.
- `POST /api/action/move` - движение по D-pad.
- `POST /api/action/transition` - подтверждение или отмена перехода.
- `POST /api/action/attack` - атака выбранной цели.

## Главные проблемы

- Сейчас web-клиент собран слишком монолитно: `index.html`, `app.js` и `style.css` фактически обслуживают всё сразу. Для полноценного сайта это нужно разделить на модули/экраны.
- Нет устойчивой клиентской сессии. При перезагрузке телефон может потерять состояние и вернуться к первому экрану.
- Текущий вход сразу создаёт персонажа через `CampaignGameSession.AddCharacterPlayer`, а по новой логике сначала должен появляться подключённый игрок без персонажа.
- Лобби кампании пока не рассчитано на состояние "игрок подключён, но персонажа ещё нет".
- Игроки на телефоне пока не участвуют в выборе/ожидании кампании вместе с GM.
- Телефонная атака сейчас не использует новую боевую систему. Она напрямую снимает `1 HP`, обходя оружие, способности, защитный бросок, броню, статусы и очередь хода.
- Движение с телефона должно идти через те же правила, что и движение в бою у GM: проверка активного хода, очков движения и доступной зоны.
- Клиент не показывает важное боевое состояние: броню, броски, активное оружие, доступные атаки, защиту, статусы и чей сейчас ход.
- На телефоне нет нормального боевого лога.
- В части файлов уже есть битая кириллица, её нужно отдельно вычистить.
- QR-код сейчас грузится через внешний сервис. Для стабильности лучше генерировать QR локально или иметь fallback с текстовой ссылкой.

## Правильная архитектура

Телефон не должен сам считать бой. Он отправляет только намерение:

- "хочу сдвинуться на клетку";
- "хочу атаковать цель";
- "хочу сменить оружие";
- "хочу закончить ход";
- "хочу подтвердить переход".

Unity проверяет правила и возвращает новое состояние. Это важно, чтобы телефон не мог сломать бой и чтобы GM, player view и мобильный интерфейс видели одну и ту же правду.

Для бросков кубика телефон должен отправлять не итог атаки, а подтверждённый результат броска. Unity хранит ожидаемое действие и применяет его только после ответа игрока.

Пример:

1. Игрок выбирает цель.
2. Unity создаёт ожидание `attack_roll`.
3. На телефоне появляется виджет кубика.
4. Игрок жмёт на кубик.
5. Телефон показывает результат, кнопки "Отправить" и "Перебросить".
6. После отправки Unity выбирает способность по выпавшему числу и продолжает атаку.
7. Если цели нужна защита игрока, Unity создаёт ожидание `defense_roll`.
8. Телефон защищающегося игрока показывает кубик защиты.
9. Только после отправки защитного броска Unity считает итоговый урон.

## Что должен показывать телефон в бою

- Имя и портрет своего персонажа.
- HP и броню: `CurrentHp / MaxHp`, `CurrentArmor / MaxArmor`.
- Очки движения.
- Броски: `CurrentRolls / MaxRolls`.
- Количество доступных ходов/действий в футере.
- Активное оружие.
- Кнопку смены оружия.
- Список доступных атак активного оружия.
- Ближайшие или доступные цели.
- Активные статусы.
- Информацию, чей сейчас ход.
- Боевой лог последних событий.
- Виджет броска кубика, когда от игрока требуется бросок атаки или защиты.
- Кнопки "Отправить" и "Перебросить" после броска.

## Что должен уметь телефон в бою

- Двигать свою фишку только в свой ход и только в пределах доступных очков движения.
- Выбирать цель из доступных целей.
- Запрашивать атаку, чтобы Unity провела полный расчёт боя.
- Сменить активное оружие.
- Закончить ход.
- Видеть результат атаки: бросок, выбранная способность, защита, броня, HP, статусы.
- Открывать настройки персонажа.
- Открывать инвентарь.
- Настраивать способности оружия, если игрок получил уровень или хочет поменять сборку.

## Что должен уметь телефон вне боя

- Показывать состояние кампании и лобби.
- Показывать, к какому игроку привязан персонаж.
- Позволять создать персонажа.
- Позволять импортировать существующего персонажа.
- Позволять открыть настройки персонажа.
- Позволять открыть инвентарь.
- Показывать кнопку камеры.

Кнопка камеры нужна в обычной кампании. Она должна подсвечиваться, если камера сейчас сфокусирована на этом игроке. При нажатии камера на player view переключается на фишку этого игрока. Это нужно, если игроки находятся в разных локациях.

## Лобби и готовность

В лобби должны быть разные состояния игрока:

- подключён, но персонажа нет;
- создаёт персонажа;
- импортирует персонажа;
- персонаж выбран, но игрок не готов;
- игрок готов;
- игрок отключён.

На player view и в лобби GM портрет без персонажа отображается серым. Когда персонаж выбран, карточка становится активной.

GM не должен запускать игру/бой, пока обязательные игроки не готовы.

## Редактор персонажа на телефоне

Редактор персонажа должен быть полноценным, но адаптированным под телефон:

- базовая информация;
- характеристики;
- HP/броня/ресурсы;
- оружие;
- способности оружия;
- атрибуты;
- инвентарь;
- подтверждение готовности.

Импорт существующего персонажа должен быть отдельным экраном. После импорта нужно решить, какой портрет использовать:

- фото игрока;
- портрет персонажа;
- выбрать вручную.

## Что нужно поменять в сервере

### `/api/game/state`

Расширить ответ:

```json
{
  "player": {
    "id": "player_1",
    "name": "Имя",
    "hp": 8,
    "maxHp": 10,
    "armor": 3,
    "maxArmor": 5,
    "movement": 2,
    "maxMovement": 3,
    "rolls": 1,
    "maxRolls": 1,
    "isMyTurn": true,
    "activeWeapon": "Меч",
    "attacks": [],
    "defenses": [],
    "statuses": []
  },
  "turn": {
    "round": 2,
    "activeTokenId": "runtime_id",
    "activeTokenName": "Имя"
  },
  "targets": [],
  "log": []
}
```

Также в состоянии должен быть блок ожидания броска:

```json
{
  "pendingRoll": {
    "id": "roll_123",
    "type": "attack",
    "dice": "d6",
    "canReroll": true,
    "rerollCost": null,
    "sourceTokenId": "runtime_id",
    "targetTokenId": "runtime_id"
  }
}
```

### `/api/action/move`

Не менять позицию напрямую. Передавать команду в боевую/кампанийную логику и получать отказ, если ход невозможен.

### `/api/action/attack`

Не снимать HP напрямую. Должен вызывать общий боевой метод, который уже считает:

- бросок атаки;
- способность оружия;
- урон от характеристики и коэффициента оружия;
- коэффициент способности;
- защитный бросок цели;
- защитную способность;
- броню;
- HP;
- атрибуты атаки и защиты;
- возможную контратаку или дополнительные ходы.

### Новые API

- `GET /api/session/restore` - восстановить телефон после перезагрузки страницы.
- `POST /api/session/register` - зарегистрировать игрока без создания персонажа.
- `GET /api/lobby/state` - состояние лобби, кампании и готовности игроков.
- `GET /api/campaigns` - список кампаний, если игрокам нужно видеть выбор вместе с GM.
- `POST /api/character/import` - привязать существующего персонажа.
- `POST /api/character/create` - создать нового персонажа.
- `POST /api/character/update` - сохранить изменения редактора персонажа.
- `POST /api/player/ready` - отметить игрока готовым.
- `POST /api/action/switch-weapon` - сменить активное оружие.
- `POST /api/action/end-turn` - закончить ход.
- `POST /api/action/request-attack` - запросить атаку и создать ожидание броска.
- `POST /api/roll/submit` - отправить результат броска.
- `POST /api/roll/reroll` - запросить переброс.
- `POST /api/camera/focus` - переключить камеру на игрока.
- `GET /api/game/log?playerId=...` - если лог станет большим и его не нужно гонять в каждом state.

## План работ

1. Разнести web-клиент на нормальную структуру файлов.
2. Починить текст и кодировку web-клиента.
3. Добавить устойчивую сессию телефона после перезагрузки страницы.
4. Разделить регистрацию игрока и создание персонажа.
5. Добавить состояние лобби: подключён, без персонажа, создаёт, импортирует, готов.
6. Показать игрокам лобби кампании вместе с GM.
7. Добавить импорт персонажа.
8. Добавить мобильный редактор персонажа.
9. Расширить `/api/game/state` под новую боевую модель.
10. Переделать `/api/action/move`, чтобы он проверял ход и очки движения.
11. Переделать атаку на двухфазную схему: запрос атаки, бросок игрока, отправка результата, расчёт Unity.
12. Добавить защитный бросок на телефон защищающегося игрока.
13. Добавить на телефон карточку персонажа: HP, броня, броски, движение, оружие, статусы.
14. Добавить выбор атаки и смену оружия.
15. Добавить кнопку завершения хода.
16. Добавить инвентарь и настройки персонажа.
17. Добавить кнопку фокуса камеры.
18. Добавить боевой лог.
19. Убрать зависимость QR от внешнего сервиса или добавить резервный вариант.

## MVP

Минимально полезная версия:

- нормальный вход по QR;
- восстановление сессии после перезагрузки страницы;
- отдельная регистрация игрока без немедленного создания персонажа;
- лобби с серыми портретами игроков без персонажей;
- выбор создать/импортировать персонажа;
- карточка персонажа на телефоне;
- отображение HP, брони, бросков, движения и активного оружия;
- движение только по правилам боя;
- атака через новую боевую систему с броском кубика на телефоне;
- защитный бросок на телефоне, когда игрока атакуют;
- кнопка завершения хода;
- кнопка смены оружия;
- кнопка камеры вне боя;
- простой лог результата атаки.

После этого телефон уже можно использовать как реальный контроллер игрока, а не только как тестовый пульт движения.
