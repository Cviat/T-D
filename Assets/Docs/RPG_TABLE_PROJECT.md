# RPG Table

## Короткое описание

RPG Table - приложение для проведения настольных ролевых игр с цифровым игровым полем. Проект не привязан к D&D или другой конкретной системе правил: мастер сам создает игровые системы, карты, персонажей, монстров, способности и сценарии.

Главная идея - дать мастеру удобный инструмент для ведения игры за столом, а игрокам показывать только нужную им часть информации: карту, видимые токены, эффекты, туман войны, лист персонажа и доступные действия.

## Цель проекта

Создать простой и гибкий инструмент для настольных RPG, который:

- уменьшает объем ручной рутины для мастера;
- не заставляет игроков изучать сотни страниц правил;
- поддерживает авторские игровые системы;
- сохраняет живое общение, импровизацию и свободу повествования;
- может работать локально за игровым столом без сложной инфраструктуры.

## Продуктовые принципы

- Мастер контролирует правду мира. Автоматизация помогает, но не диктует правила.
- Игроки видят только то, что должны видеть их персонажи.
- Система правил должна быть настраиваемой, а не зашитой в код.
- Карты способностей должны быть понятнее таблиц и длинных правил.
- MVP должен сначала работать локально, затем расширяться до веб-клиентов.
- Редактор и игровой режим должны использовать одни и те же данные карты.

## Основные режимы

### Экран мастера

Рабочее место мастера. Показывает полную информацию:

- вся карта;
- скрытые зоны и скрытые объекты;
- персонажи игроков;
- монстры и NPC;
- инициатива;
- события боя;
- свойства токенов;
- эффекты на карте;
- быстрые инструменты редактирования сцены.

### Экран игроков

Второй монитор или телевизор по HDMI. Показывает только публичную информацию:

- открытую часть карты;
- видимые токены игроков, NPC и монстров;
- эффекты, которые должны видеть игроки;
- туман войны;
- минимальный UI без мастерских инструментов.

### Веб-клиент игрока

Телефонный клиент без установки приложения. Игрок подключается по QR-коду и получает:

- своего персонажа;
- инвентарь;
- карты способностей;
- журнал;
- локальную карту или мини-карту;
- возможность отправлять мастеру заявки на действия.

На первом этапе веб-клиент не обязателен для MVP. Его лучше проектировать как будущий слой поверх уже работающего локального ядра.

### Редактор карт

Инструмент создания игровых полей:

- тайловая карта;
- объекты окружения;
- слои видимости;
- стартовые позиции;
- зоны;
- интерактивные объекты;
- сохранение и загрузка карт.

### Карточная система способностей

Действия персонажей описываются картами. Карта - это удобная форма способности, предмета, заклинания, маневра или особого действия.

Карта может содержать:

- название;
- описание;
- стоимость;
- дальность;
- область действия;
- цель;
- условия применения;
- эффекты;
- визуальный эффект на карте;
- текст для игрока;
- скрытые мастерские параметры.

## Первое MVP

Цель MVP - получить минимально полезный цифровой стол, который уже можно показать и потрогать в Unity.

### Входит в MVP

- 2D карта на Tilemap или сетке.
- Токены персонажей и монстров.
- Перетаскивание токенов мышью на экране мастера.
- Два режима отображения: Master View и Player View.
- Простая система скрытия объектов от игроков.
- Простой туман войны: открытые и закрытые клетки.
- Базовая инициатива: список участников и текущий ход.
- Черновая карточка способности как ScriptableObject.
- Несколько тестовых карт способностей.
- Сохранение тестовых данных в ассетах Unity.

### Не входит в первый MVP

- Полноценный веб-клиент.
- Авторизация игроков.
- Сложная сеть.
- Генератор правил.
- Полный редактор кампаний.
- Красивый production UI.
- Сложная автоматизация боевых правил.

## Предлагаемая архитектура Unity

### Runtime-слои

- `Core` - чистые модели данных: персонаж, токен, карта, способность, эффект, инициатива.
- `Board` - сетка, клетки, координаты, слои карты, туман войны.
- `Presentation` - визуальное отображение токенов, тайлов, эффектов и UI.
- `Input` - мышь, клавиатура, выбор объектов, drag and drop.
- `GameMaster` - инструменты мастера и команды управления сценой.
- `PlayerView` - фильтрованное отображение публичного состояния.
- `Persistence` - сохранение и загрузка данных.

### Данные

На раннем этапе разумно использовать ScriptableObject для:

- шаблонов персонажей;
- шаблонов монстров;
- способностей;
- тайлсетов;
- карт;
- игровых систем.

Позже можно добавить JSON-сохранения для кампаний и пользовательского контента.

### Важное разделение

Состояние игры не должно жить только в GameObject-ах сцены. GameObject должен быть представлением данных, а не единственным источником правды.

Правильная модель:

- данные: `TokenData`, `BoardData`, `AbilityCardData`;
- визуал: `TokenView`, `BoardView`, `AbilityCardView`;
- команды: `MoveTokenCommand`, `RevealCellsCommand`, `ApplyAbilityCommand`.

## Риски и решения

### Два экрана

Unity умеет работать с несколькими дисплеями, но это надо проверять рано. Для MVP можно сначала сделать переключаемые камеры или два Canvas-режима, а затем добавить настоящий вывод на второй монитор.

### Веб-клиент

Веб-клиент - отдельная большая подсистема. Ее лучше не делать первой. Сначала нужно стабилизировать модель состояния игры и команды. После этого можно думать о локальном сервере, QR-коде и синхронизации.

### Настраиваемые правила

Не стоит сразу делать универсальный визуальный редактор правил. На первом этапе достаточно карточек способностей с простыми типами эффектов: урон, лечение, перемещение, статус, зона.

### Туман войны

Первую версию лучше делать клеточной: клетка открыта или закрыта для игроков. Более сложная видимость по стенам, источникам света и линии обзора может появиться позже.

## Первый технический вертикальный срез

1. Создать структуру папок `Assets/RPGTable`.
2. Создать сцену `RPGTablePrototype`.
3. Сделать тестовую 2D сетку.
4. Добавить 2-3 токена.
5. Реализовать выбор и перетаскивание токена.
6. Добавить флаг `VisibleToPlayers`.
7. Сделать переключение Master/Player View.
8. Добавить простую панель инициативы.
9. Добавить ScriptableObject для карты способности.
10. Показать одну карту способности в UI.

## Открытые вопросы

- Игра должна быть строго 2D сверху или допускается 2.5D/изометрия?
- Карта должна быть квадратной, гексагональной или поддерживать оба варианта?
- Основная платформа мастера: Windows build, Unity Editor, WebGL или все сразу?
- Игроки за столом будут управлять действиями сами или только отправлять заявки мастеру?
- Нужна ли обязательная работа без интернета?
- Должны ли правила уметь считать броски кубов, или мастер вводит результат вручную?
- Насколько важна поддержка импорта карт из изображений?
- Нужен ли в первом прототипе второй физический монитор, или достаточно переключателя Master/Player View?

## Текущее решение для старта

Начинаем с локального Unity-прототипа для мастера:

- 2D top-down;
- квадратная сетка;
- один Unity build/Editor;
- без сети;
- без веб-клиента;
- данные через ScriptableObject;
- визуальный прототип важнее финальной архитектурной полноты.

После появления рабочего стола с токенами, туманом войны и инициативой возвращаемся к веб-клиенту и синхронизации.
