# План боевого UI

Цель: убрать боевой интерфейс из runtime-сборки через `new GameObject` и перевести его на редактируемые prefab-ы.

## Общие правила

- Runtime-код не должен рисовать сложный UI вручную.
- Runtime-код должен загружать prefab, создавать instance и передавать данные во View-компонент.
- Визуальный вид должен настраиваться в Unity через prefab.
- Если prefab отсутствует, система должна не падать, а явно сообщать в лог.

## Prefab-ы

Планируемые prefab-ы:

- `TokenWorldBars.prefab` - HP/armor бары над фишкой.
- `CombatInspector.prefab` - боевой инспектор выбранной фишки.
- `CombatTurnBar.prefab` - верхняя очередь ходов.
- `CombatTurnCard.prefab` - карточка участника в очереди.
- `CombatLogPanel.prefab` - панель боевого лога.
- `CombatLogEntry.prefab` - строка боевого лога.
- `CombatStatusIcon.prefab` - иконка активного статуса.

## Этап 1. Бары над фишками

Сделать первым:

- `TokenWorldBarsView`.
- `TokenWorldBars.prefab`.
- Подключить `TokenHealthArmorBars` к prefab-у.
- Убрать создание HP/armor bars вручную из runtime.

Данные:

- `CurrentHp / MaxHp`
- `CurrentArmor / MaxArmor`
- видимость только в бою и только на живой фишке

## Этап 2. Боевой инспектор

Новый инспектор должен показывать:

- HP.
- Броню.
- Очки движения.
- Очки бросков.
- Активное оружие.
- Атаки активного оружия.
- Защитные слоты.
- Активные статусы.

## Этап 3. Очередь ходов

Очередь должна брать данные из `CombatManager.Queue`, а не сортировать токены по имени.

Нужно показывать:

- текущего участника;
- следующих участников;
- номер раунда;
- состояние HP/armor;
- мертвые токены отдельно или скрывать.

## Этап 4. Боевой лог

Лог должен показывать цепочку боя:

```text
Атака: roll 4 -> Удар x1.5
Защита: roll 2 -> Блок 4
Броня: -3
HP: -5
Атрибут: Оглушение
```

## Этап 5. Удаление старого runtime UI

После подключения prefab-ов удалить или упростить старые runtime-генераторы:

- ручной turn bar;
- ручные banner-объекты, если появится prefab-аналог;
- старый слабый combat inspector.
